Howl host-renderer integration contract

1. Product shape

- the product is an embeddable terminal widget, not a window-owning terminal app.
- hosts own layout, focus, input routing, composition, and present timing.
- `howl-term` owns terminal runtime, widget invalidation, and the retained render surface.
- renderer is a projector that maintains one cached terminal image and updates it only when stale.

2. Singular path for current codebase

- current implementation chooses one path only: renderer-owned render surface.
- host-owned target textures are out of scope for the current codebase.
- future host-owned target support may be added later as an alternate integration mode, not a co-equal path today.
- naming should stay neutral enough that later surface import is possible without a rewrite.

3. Ownership split

- `vt-core` owns canonical terminal-visible state.
- `render-core` should consume terminal-visible state as directly as practical.
- `howl-term` owns redraw truth, render scheduling, and surface lifecycle.
- hosts must not compute dirty heuristics and must not own terminal pixel truth.

4. Surface contract

- each terminal widget owns one retained render surface.
- the surface is exported to hosts as a native handle suitable for composition.
- hosts consume the exported handle; they do not write terminal pixels directly.
- surface lifetime, resize, and recreation are owned by the widget/runtime.
- if a surface handle changes, the widget/runtime must expose the new handle explicitly.

5. Rendering model

- rendering is event-driven, not tied to an unconditional frame loop.
- if terminal-visible state is unchanged, renderer work should be skipped entirely.
- the last completed surface contents remain valid while idle and may be presented repeatedly by the host.
- multiple invalidations may coalesce into one redraw of the latest state.

6. Invalidation model

- terminal-visible mutation marks widget state stale.
- resize marks the visible surface stale.
- cursor blink or other animated terminal-visible states are timer-driven invalidations, not fixed-FPS rendering.
- dirty state is acknowledged only after a successful draw/present boundary.
- failed render/present must not clear dirty state.

7. Host scheduling model

- hosts wait on event-driven wake sources, not on a tight render loop.
- wake sources include PTY output, input affecting the visible state, resize, and widget-owned animation ticks.
- on wake, hosts drain their platform event queue, then ask the widget whether redraw work exists.
- if redraw work exists, hosts request one render update and present the latest exported surface.
- if only host chrome changed, hosts may present without forcing a terminal redraw.

8. Current implementation direction

- remove per-frame host texture injection from the main rendering path.
- let each terminal instance own and export its render texture.
- keep the codebase singular around renderer-owned surfaces until a real integration forces host-provided targets.

9. Non-negotiable invariant

- redraw truth is widget-owned and host-agnostic.
- hosts decide when work runs and how surfaces are composed, never what terminal pixels changed.
